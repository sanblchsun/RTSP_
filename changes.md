# Изменения в проекте RTSP_

## Проблема 1: Пульсация качества каждые ~2 секунды

**Причина:** В `WinRT-API/encoder_x264.cpp:56` был `i_keyint_max = fps * 2 = 60` (IDR каждые 2 сек) + `i_scenecut_threshold = 0` (нет адаптивных IDR) + `b_intra_refresh = 0`.

**Исправление (`WinRT-API/encoder_x264.cpp`):**

1. **`b_intra_refresh = 1`** (строка 61) — вместо одного большого IDR-кадра intra-макроблоки распределяются по всем кадрам равномерно. Битрейт становится гладким, пульсация качества полностью исчезает.

2. **`i_keyint_min = fps`** (строка 57) — корректный минимум цикла обновления для intra_refresh (было 1).

3. **`b_aud = 0`** (строка 75) — отключены AU Delimiter NAL'ы, которые добавляли лишний overhead к каждому кадру (было 1).

4. **SPS/PPS без start code** (строки 120, 122) — `assign(p + offset, p + sz)` вместо `assign(p, p + sz)`. Раньше SPS/PPS сохранялись вместе с annex-B start code (0x00 0x00 0x00 01). Из-за этого `rtsp_server.cpp:447` кодировал их в base64 для `sprop-parameter-sets` в SDP **со стартовыми байтами**, что не соответствует RFC 3984.

## Проблема 2: Задержка при VPS (без UDP)

**Исправление (`webrtc_server.py`):**

1. **TCP_NODELAY на сокете агента** (строка 369) — `conn.setsockopt(socket.IPPROTO_TCP, socket.TCP_NODELAY, 1)` отключает алгоритм Нейгла на принятом соединении от агента. Без этого мелкие RTP-пакеты задерживались на ~200 мс перед отправкой. На стороне C++ агента `TCP_NODELAY` уже был установлен (`main.cpp:86`).

2. **Очередь уменьшена с 60 до 5 кадров** (строка 110) — `asyncio.Queue(maxsize=5)`. Раньше буфер накапливал до 2 секунд видео (60 кадров @ 30fps). Теперь ~0.17 сек. Если очередь переполняется — новый кадр отбрасывается (`_put` дропает при QueueFull, строка 151), что лучше, чем растущая задержка.

3. **SPS/PPS для WebRTC signalling** (строки 123-132, 434-444) — добавил извлечение SPS/PPS из входящего H.264 потока в `H264StreamTrack.feed_nal()` и инжекцию в answer SDP в `/offer`:
   - Когда приходит NAL с типом 7 (SPS) или 8 (PPS), он сохраняется в глобальные `_sps_data`/`_pps_data` (thread-safe, через `_sps_pps_lock`)
   - В `/offer` после создания ответа SDP, если SPS/PPS доступны, в строку `a=fmtp:96` дописывается `;sprop-parameter-sets=Base64(SPS),Base64(PPS)`
   - **Это критически важно** при `b_intra_refresh = 1`: без IDR-кадров у браузера нет другого способа получить SPS/PPS (они больше не повторяются в потоке). Без этой правки браузер показывал бы чёрный экран.

**Исправление (`main.cpp`):**

4. **Frame pacing** (строки 315-316, 319, 381-384) — добавил тайминг main-цикла:
   - `frame_start = steady_clock::now()` в начале цикла (строка 319)
   - `frame_duration = 1000/kFps ms ≈ 33ms` (строка 316)
   - После отправки: `sleep_for(frame_duration - elapsed)` (строка 383-384)
   - Раньше цикл захватывал/кодировал/отправлял на максимальной скорости (60+ FPS). Теперь стабильные 30 FPS. RTP timestamp step (`90000/kFps`) синхронизирован с реальным темпом отправки.

5. **`const int kFps = 30`** (строка 217) — единая константа FPS, используется во всех 5 местах: capture (строка 242), encoder (строка 254), rtsp (строка 301), rtp_timestamp_step (строка 315), frame_duration (строка 316).

**Исправление (`rtsp/rtsp_server.cpp`):**

6. **recv читает блоками, а не по 1 байту** (строка 190-191) — `recv(fd, buf, sizeof(buf)-1-total_read, 0)` вместо `recv(fd, buf + total_read, 1, 0)`. RTSP-запросы читаются за 1-2 вызова вместо сотен.

## Проблема 3: Найденные ошибки

1. **SPS/PPS со start code** (`encoder_x264.cpp:120,122`) — описано выше в Проблеме 1, пункт 4.

2. **FPS был жёстко зашит числом 30** (`main.cpp:313` и ещё 3 места) — теперь `const int kFps` (строка 217).

3. **FU-A: потеря фрагментов при перепутанных пакетах** (`webrtc_server.py:200-203`) — если приходил `start=1` при уже начатом FU-A (out-of-order), старый буфер молча перезатирался. Добавлено предупреждение: `"FU-A: new frame started before previous finished (dropped)"`.

4. **RTSP сервер читал по 1 байту** (`rtsp_server.cpp:190`) — исправлено на чтение блоками.

5. **Нет TCP_NODELAY на Python сервере** (`webrtc_server.py:369`) — добавлен.

6. **Очередь 60 кадров** (`webrtc_server.py:110`) — уменьшена до 5, убирает 2 секунды буферизации.

7. **Нет sprop-parameter-sets в WebRTC SDP** (`webrtc_server.py:434-444`) — добавлен.

## Раунд 2: Дополнительные оптимизации задержки

### (`WinRT-API/encoder_x264.cpp:68-69`): CRF → CQP

**Было:** `X264_RC_CRF` с `f_rf_constant = qp` и `f_rf_constant_max = qp + 3`. CRF варьирует QP от кадра к кадру — размер пакета скачет, TCP send buffer скачет, задержка неравномерная.

**Стало:** `X264_RC_CQP` с `i_qp_constant = qp`. QP фиксирован на каждом кадре. Размер пакета зависит только от сложности сцены, не от номера кадра. TCP поток гладкий → предсказуемая задержка. На статичном десктопе битрейт падает сам (меньше данных), без скачков QP.

### (`main.cpp:125-146`): Фикс частичной отправки в `SendRtp()`

**Было:** `send()` вызывался один раз, возврат проверялся на `<= 0`. Если `send()` отправлял меньше байт, чем запрошено (может происходить при любой сетевой нагрузке), interleaved framing ломался — все последующие пакеты уходили со сдвигом.

**Стало:** Добавлен `send_all()` — цикл досылает остаток, пока все байты не отправлены или пока `send()` не вернёт ошибку. Используется для заголовка и данных.

### (`main.cpp, webrtc_server.py, encoder_x264.h/cpp`): RTCP PLI/FIR → keyframe forwarding

**Проблема:** При переключении вкладок в браузере или при новом подключении aiortc получает RTCP PLI/FIR и вызывает `_request_keyframe()` на `VideoStreamTrack`. Без обработки — браузер ждёт ~2 секунды, пока intra_refresh обновит всю картинку.

**Исправление (`webrtc_server.py`):**
- `H264StreamTrack` переопределяет `_request_keyframe()` — при вызове от aiortc отправляет callback в основной цикл
- Callback регистрируется в `lifespan` и вызывает `AgentSession.request_keyframe()`
- `AgentSession.request_keyframe()` шлёт сигнал `!K\r\n` по TCP агенту
- Глобальные `_agent_session`/`_agent_lock` для thread-safe доступа к активной сессии

**Исправление (`main.cpp`):**
- `RtspClient.CheckForIncoming()` — неблокирующий `select()` на сокете. Если есть данные и они содержат `!K`, устанавливает флаг `keyframe_requested_`
- В главном цикле `push_mode`: перед отправкой RTP вызывается `CheckForIncoming()`, при флаге — `encoder->RequestKeyframe()`

**Исправление (`WinRT-API/encoder_x264.cpp`):**
- `X264Encoder::RequestKeyframe()` — вызывает `x264_encoder_intra_refresh(m_encoder)` для сброса цикла intra refresh. Следующие кадры обновляют макроблоки с начала, браузер быстрее получает полную картинку.
